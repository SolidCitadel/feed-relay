# ADR-0006: 모듈 간 통신 2평면 — 데이터=오케스트레이션 직접 호출, 통지=애플리케이션 이벤트

- 날짜: 2026-07-03 / 상태: Accepted

## 맥락

"MSA 지향이면 이벤트 기반이어야 하지 않나"라는 도전에 대해, 오케스트레이션/코레오그래피(로직 위치)와 동기/비동기(전송)가 독립 축임을 확인. 파이프라인(fetch→evaluate→diff→apply)은 매 단계 반환값이 다음 입력이고, 결과가 소유자(Run)에게 돌아와야 멱등성 불변식(SyncMapping 기록)이 성립한다.

## 검토한 대안

1. **전면 코레오그래피**: 모든 요청-응답은 이벤트 쌍으로 환원 가능하나, 조립 책임이 사라지지 않고 correlation 상태 기계로 형태만 바뀜(sync는 여전히 모든 완료 이벤트를 runId로 조립). 결합은 포트 의존→이벤트 타입 의존으로 이름만 변경. 추가 비용: 홉당 트랜잭션 분리(최종 일관성), 전 핸들러 멱등성 의무, 흐름 관측 곤란 — 단일 프로세스에서 이득(독립 배포·자율 팀·버스트 흡수) 전무. Netflix Conductor가 순수 코레오그래피의 실패에서 나온 물건이라는 산업 증거.
2. **전면 직접 호출**(초기안): ConnectionRevoked(위임 철회→구독 정지)는 sync→connections 의존 방향상 역방향 직접 호출이 순환 — 성립 불가. RunFailed→알림도 러너가 알림의 존재를 알아야 할 이유 없음.
3. **2평면 혼재** (채택)

## 결정

- 데이터 평면(결과값이 호출자의 분기·상태 변경에 쓰임) = 직접 호출. 흐름 전체 불변식+소유자(Run)가 있으므로 오케스트레이션.
- 통지 평면(발행 후 발행자 분기 불변) = `@ApplicationModuleListener`(커밋 후 비동기, at-least-once). v1: `RunCompleted`·`RunFailed`(sync 발행→notifications), `ConnectionRevoked`(connections 발행→sync). 이벤트 타입은 발행자 api 소속. notifications 모듈 신설.
- 판별 테스트(02 등재): ①제2의 무관한 리스너 상상 가능? ②다음 분기가 결과 의존? ③흐름 불변식+소유자 존재?

## 결과

- 혼재는 절충이 아니라 통신의 두 종(명령/사실)에 각 표준을 적용한 것 — 사가 문헌(오케스트레이션/코레오그래피 양분), EIP(Command/Event Message), Modulith 자체 설계(모듈 API 호출 + 크로스 모듈 이벤트)와 정합.
- 분리 시: 직접 호출 → 오케스트레이티드 사가(전송만 교체), 통지 이벤트 → Modulith externalization으로 브로커 승격.
