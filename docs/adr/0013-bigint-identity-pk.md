# ADR-0013: 전 테이블 공통 PK는 bigint identity

- 날짜: 2026-07-04 / 상태: Accepted

## 맥락

M2의 첫 Flyway 마이그레이션(users)이 PK 타입 결정을 강제했다. §8.3 ERD는 `id PK`로 타입 미정이었고, 이 선택은 전 테이블·전 FK 인덱스에 파급되며 사후 전환 비용이 크다. 결정 기준은 "현재 규모에서의 성능"이 아니라 **"어떤 확장 경로가 이 선택을 깨서 이전을 강제하는가"**로 잡았다.

## 검토한 대안

1. **bigint identity** (채택): DB 시퀀스 생성, 8바이트, B-tree 오른쪽-끝 삽입으로 지역성 최적. 운영 가독성(로그 대조·수동 SQL·짧은 URL — 품질 목표 "운영 단순성") 우위. 약점: 순차 id의 enumeration(규모 추정) — 단 API 전체가 세션 인증+소유권 검사 뒤에 있고, 공개 노출 리소스가 생기면 해당 리소스에만 public token 칼럼을 추가하는 국소 해법이 존재.
2. **UUIDv7**: 16바이트, 타임스탬프 상위비트로 삽입 지역성은 bigint 근접. 앱/DB(PG18+ `uuidv7()`) 생성 가능, 쓰기 분산에 안전, enumeration 무대응. 약점: FK 인덱스 2배, 로그·수동 SQL 가독성, 타임스탬프 노출(생성 시각 유출). 이기는 축(분산 생성·INSERT 전 id 확보)이 현 아키텍처 계획에 수요가 없음.
3. **UUIDv4**: 완전 랜덤 삽입으로 B-tree 지역성 최악. v7 대비 우위가 완전 익명성뿐 — 기각.

## 확장 경로 분석 (결정 근거)

| 확장 경로 | bigint가 깨지는가 |
|---|---|
| 사용자·데이터 증가 (동일 아키텍처) | 아니오 — 상한 9.2×10¹⁸, 단일 PG에서 인덱스 우위 유지 |
| 앱 멀티 인스턴스·읽기 레플리카 | 아니오 — 쓰기 DB 단일, 시퀀스가 동시성 처리 |
| MSA 분리 | 아니오 — 테이블 소유권이 모듈 단위(§8.3)라 테이블+시퀀스가 서비스와 함께 이동 |
| 동일 테이블 다중 쓰기 노드 (샤딩·멀티 프라이머리·클라이언트 오프라인 생성) | **예 — 유일한 파괴 경로** |

유일한 파괴 경로는 "성장의 연속선"이 아니라 쓰기 모델의 아키텍처 피벗이며, 그 시점엔 단일 PG 전제의 나머지(잡큐 `FOR UPDATE SKIP LOCKED`·트랜잭션 경계·이벤트 레지스트리)가 함께 재설계된다(ADR-0001의 트리거 층위). id 전략은 그 재설계의 일부다.

## 결정

- 전 테이블 PK = `bigint GENERATED ALWAYS AS IDENTITY`. FK도 bigint.
- JPA IDENTITY 전략 사용(insert batching 비활성 부작용은 항목 단위 처리 파이프라인이라 실질 무영향 — 필요 시 bigint 유지한 채 SEQUENCE 전략 전환 가능).

## 결과

- 재검토 트리거 ①: 공개 노출 리소스(공유 링크 등) 등장 → 해당 리소스에만 public token 칼럼 추가 (전면 전환 불필요).
- 재검토 트리거 ②: 동일 테이블 다중 쓰기 노드 수요 → UUIDv7 재검토 (PG18+ 네이티브 `uuidv7()` 전제).
- 트레이드오프: enumeration 내성·INSERT 전 id 확보를 포기 — 각각 국소 해법·수요 부재로 수용.
