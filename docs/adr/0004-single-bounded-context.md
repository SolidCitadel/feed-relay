# ADR-0004: 단일 바운더리 컨텍스트 + 서브도메인 정렬 모듈

- 날짜: 2026-07-02 / 상태: Accepted

## 맥락

초기 설계는 6개 패키지를 "바운더리 컨텍스트"로 명명했으나, 검증 과정에서 판별 기준(모델·언어의 일관성 경계)을 정면 적용한 결과 오류로 판정. 일반명사("계정", "항목", "목적지")의 자연스러운 다의성은 경계 증거가 아니며, 발견된 모든 모호성이 **용어 신조**(User/Connection, SourceItem/OutboundItem, Slot/SlotMapping/TargetRef)로 해소됨 — 신조로 해소되면 언어는 하나다.

## 검토한 대안

1. 컨텍스트 6개(초기안): 근거였던 "확장 계획이 경계선을 따라 갈라짐"은 모듈(변경 축) 논리지 컨텍스트(의미) 논리가 아님 — 범주 오류로 기각
2. **단일 컨텍스트 + 모듈 6(+1)개** (채택)

## 결정

FeedRelay는 단일 바운더리 컨텍스트(= docs/02 사전 하나가 전체 언어). 진짜 언어 경계는 소유하지 않은 외부 언어 2곳(ical 세계 ↔ ingestion, Google Tasks 세계 ↔ delivery)뿐이며 어댑터가 번역(ACL). 내부 분할은 서브도메인(core: rules·sync / supporting: ingestion·delivery / generic: identity·connections·notifications)을 따르는 **모듈**이다(Evans MODULES: 도메인 개념 정렬 + 응집/결합). 컨텍스트와 모듈의 관계는 등식이 아니라 한 방향 제약("모듈이 두 컨텍스트에 걸치면 안 됨")뿐.

## 결과

- "바운더리 컨텍스트"를 내부 6분할 지칭으로 쓰는 것 금지(02 폐기 용어).
- 두 번째 컨텍스트 생성 조건 = 개명으로 해소되지 않는 용어 충돌(별도 팀, 인수 레거시)이며, 모듈 비대화는 조건이 아니다.
