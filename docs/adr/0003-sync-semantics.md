# ADR-0003: 동기화 의미론 — 단방향 생성+수정, 완료 후 불개입, 재라우팅 시 기존 유지

- 날짜: 2026-07-02 / 상태: Accepted

## 맥락

이 시스템의 최난제는 분류가 아니라 동기화 상태 관리(멱등성·reconciliation). LMS는 마감일 변경이 잦아 create-only는 신뢰를 빨리 잃고, 삭제 동기화는 오삭제 리스크·정책 복잡도가 급증.

## 검토한 대안

1. 생성만(fire-and-forget): 마감 변경 미반영 — 기각
2. 삭제까지 완전 동기화: tombstone 등 복잡도 급증 — 기각
3. 양방향: LMS 쓰기 불가로 성립하지 않음 — 기각
4. **생성+수정 단방향** (채택)

파생 정책: 대상 앱에서 완료된 항목에 소스 변경 도착 시 → 재오픈(사용자 노이즈) vs 불개입(채택). 규칙 변경으로 기존 항목의 목적지가 바뀔 때 → 이동/재생성(메모·상태 유실 위험) vs 기존 위치 유지·신규부터 적용(채택).

## 결정

소스↔대상 ID 매핑(SyncMapping)을 저장하고 생성+수정만 반영. 완료·삭제 감지 시 FROZEN(사유 구분) — 이후 불개입. content_hash는 변환 후 최종 산출물(OutboundItem+TargetRef) 기준 → transform 변경은 내용만 갱신, 위치는 external_ref가 고정. 피드에서 항목 소실은 무시(삭제 동기화 없음과 일관).

## 결과

- sync_mappings 스키마(UNIQUE(subscription_id, source_uid), status, frozen_reason)가 이 결정의 직접 산물.
- 멱등성 시나리오("같은 피드 2회 → 무변화", "마감 변경 → update 1건", "완료 후 → 불개입")가 테스트 1급 시민.
- 재검토 트리거: 삭제 동기화 요구가 실사용자에게서 반복 제기될 때.
