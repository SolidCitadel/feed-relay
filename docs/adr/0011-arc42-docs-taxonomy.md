# ADR-0011: 문서 체계는 arc42 경량 채택

- 날짜: 2026-07-03 / 상태: Accepted

## 맥락

기존 docs(00-shared-understanding, 01-design, 02-ubiquitous-language)는 내용 유형이 아니라 **작업 단계**(합의하기/설계하기/용어 정의하기)에서 파생된 분류였다. "합의"와 "설계"는 내용 경계가 아니어서(프로젝트의 모든 것이 합의되고 설계되므로) 모듈 맵·결정 기록이 문서 간 중복되고 이동을 반복했다. 그릇(문서 분류)을 먼저 정의하지 않은 것이 근본 원인.

## 검토한 대안

1. **토픽 기반**(product/architecture/domain/operations + adr): 실무 최다 관행, 최소 마찰. 단 체크리스트 부재 — 그릇 경계를 스스로 정의·방어해야 하며, 그게 실패했던 지점.
2. **arc42 경량**(채택): 기존 내용이 이미 골격과 ~70% 동형(모듈 트리→§5, 실행 모델→§6, 배포→§7, ADR→§9, 용어→§12). 차집합(§10 품질 시나리오, §11 리스크)이 현재 문서의 실결손. 산업 표준 가독성·학습 전시 기준(ADR-0007과 동일 기준)에 부합.
3. full arc42(전 절 후하게): 솔로 프로젝트에 이해관계자 심사용 문서량은 과잉 — 기각.
4. 유비쿼터스 언어 문서의 외부 1급 파일 유지 + §12 포인터: 검토 과정에서 "glossary 초과 내용"(거시 동작·구조 판정)이 각각 §1·§3·§5 소유로 정리되자 초과분이 소멸 — 기각하고 §12 통합.

## 결정

`docs/arc42/01~12` 파일-절 구조로 재편. arc42 공식 가이드대로 얇은 절은 몇 줄로 유지. §9는 `../adr/` 포인터. **§12(용어집)가 유비쿼터스 언어 전체(판정 기준·용어 사전·폐기 용어)를 소유** — 단일 원천 지위와 구속 범위(코드 식별자·ADR 서술·대화)는 §12에 승계. 00·01·별도 언어 문서는 해체·소멸, 내용은 각 절이 승계. 구조 **현황**(모듈 표·의존 규칙)의 주소는 §5로 일원화. docs/ 루트 = `arc42/` + `adr/` (+ 향후 사용자 가이드 패밀리).

## 결과

- 내용의 고정 주소 확립 — 중복의 구조적 원인 제거.
- 품질 시나리오(§10)·리스크(§11)가 처음으로 주소를 얻음.
- living 문서 = `docs/arc42/*` + `docs/ubiquitous-language.md`. 편집 규율은 adr/README.
