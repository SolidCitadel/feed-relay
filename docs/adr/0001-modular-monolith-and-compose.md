# ADR-0001: 모듈러 모놀리스 + Docker Compose, k3s는 2단계 마이그레이션

- 날짜: 2026-07-02 / 상태: Accepted

## 맥락

프로젝트 목적이 "무료로 시작하는 서비스(초기 지인 스케일, 성장 시 고도화)"로 확정됨. 워크로드는 30분 주기 pull 배치 × 수십~수백 사용자로, 부하가 본질적으로 낮다. 인프라는 Oracle Cloud Always Free(A1 Flex). 학습 경험(k3s/MSA) 욕구와 운영 단순성이 충돌.

## 검토한 대안

1. **MSA + k3s**: 단일 노드 k3s는 성능 마이너스(컨트롤플레인 상주)·안정성 동일(SPOF 그대로). k8s가 안정성을 주는 메커니즘(멀티노드 페일오버)은 이 앱의 가용성 요구(10분 죽어도 동기화가 늦을 뿐)에 과잉. Postgres 등 stateful의 k8s 운영은 고난도.
2. **MQ/Redis 선도입**: 생산자=소비자=단일 프로세스인 pull 배치에 큐가 살 조건(push 버스트, 워커 분산, 서비스 간 디커플링)이 없음. 잡큐는 Postgres `FOR UPDATE SKIP LOCKED`로 충분.
3. **모듈러 모놀리스 + Compose** (채택)

## 결정

모듈러 모놀리스(Spring Modulith, 경계는 테스트로 강제) + Docker Compose + PostgreSQL. MQ 도입 트리거 = 웹훅(push) 소스 추가 또는 멀티 워커. Redis 트리거 = 멀티 인스턴스 전환. k3s는 앱 안정화 후 **멀티노드 마이그레이션을 별도 운영 프로젝트(2단계)** 로 수행 — "트리거를 정의하고 이주했다"가 경험 서사로도 우월.

## 결과

- 같은 수고의 실질 안정성 투자는 백업(오프박스)·알림·모니터링·멱등성 테스트로 향한다.
- 모듈 경계(api 계약, 테이블 비공유)가 분리 가능성의 실체 — 통신 스타일이 아니다.
- 재검토 트리거: 위 명시된 도입 트리거 발생 시.
