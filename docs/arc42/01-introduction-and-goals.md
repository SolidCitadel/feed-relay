# 1. 소개와 목표

**FeedRelay**: 다양한 소스(v1: ical)의 항목을 정규식 규칙(템플릿)으로 분류·변환해 캘린더/Todo 앱(v1: Google Tasks)에 단방향 동기화하는 서비스. 첫 슬라이스는 Canvas LMS 과제 → 과목별 Google Tasks 리스트.

무료로 시작하는 서비스(초기 본인+지인, 성장 시 고도화). 수익화·검증 심사는 성장 트리거로만 취급.

## 거시 동작 (일상어 서술 — 모든 용어의 뿌리)

- **동기화**: 주기적으로, 등록된 ical 주소에서 데이터를 받아, 사용자가 고른 규칙으로 제외·수정·분배하고, 사용자의 Google Tasks 리스트에 만들거나 고친다. 단 한 번씩만 만들고, 바뀐 만큼만 고치며, 사용자가 완료·삭제한 것은 다시 건드리지 않는다.
- **온보딩**: 사용자는 Google로 로그인하고, ical 주소를 등록하고, 템플릿을 골라 과목→리스트 대응을 정하고, Google Tasks에 쓸 권한을 위임한다.

## 제품 정책

- 동기화는 **단방향 생성+수정**: 완료·삭제된 항목 불개입(FROZEN), 규칙 변경 시 기존 항목 위치 유지·신규부터 적용 ([ADR-0003](../adr/0003-sync-semantics.md))
- 매칭 안 된 항목·미매핑 slot은 `_inbox` 기본 리스트로 + 대시보드 알림 (누락 0 원칙)
- 순수 일정성 항목(일정)은 v1 제외 — Google Calendar ical 구독이 이미 커버. 제외는 템플릿 규칙이 수행하며 과제 누락 없음 ([ADR-0012](../adr/0012-source-shape-knowledge-in-templates.md))
- 템플릿은 선택 시 사용자 소유로 복제(원본 key/version 기록), 업데이트는 신규 선택자에게만
- UI는 최소 셀프서브 온보딩(로그인 → 소스 등록 → 템플릿 → 매핑 → 완료), 정규식 편집기는 v2

## 품질 목표 (우선순위순)

1. **멱등성** — 단 한 번씩만 만들고, 바뀐 만큼만 고친다
2. **사용자 의사 존중** — 대상 앱에서의 완료·삭제를 되돌리지 않는다
3. **장애 격리** — 한 구독의 실패가 다른 구독·전체 실행에 전파되지 않는다
4. **운영 단순성** — 솔로 운영자가 무료 인프라에서 감당 가능
5. **확장 개방** — 소스/대상 추가가 어댑터 추가로 끝난다

## 이해관계자

- 개발자 겸 운영자 (솔로)
- 사용자: Canvas 계열 LMS를 쓰는 학생 (초기 지인 스케일, 최대 100명 — §2 제약)
